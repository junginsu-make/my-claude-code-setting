# 문서 공동 작성 워크플로우

이 스킬은 협업 문서 작성을 세 단계로 안내하는 구조화된 워크플로우를 제공합니다. 컨텍스트 수집, 개선 및 구조화, 독자 테스트의 세 단계를 거치며 적극적으로 안내합니다.

## 이 워크플로우를 제안할 때

**트리거 조건:**
- 사용자가 문서 작성을 언급: "문서 작성", "제안서 초안", "사양서 작성", "정리해서 작성"
- 특정 문서 유형 언급: "PRD", "설계 문서", "의사결정 문서", "RFC"
- 상당한 분량의 글쓰기 작업을 시작하는 것으로 보일 때

**초기 제안:**
문서 공동 작성을 위한 구조화된 워크플로우를 제안합니다. 세 단계를 설명합니다:

1. **컨텍스트 수집**: 사용자가 관련 맥락을 제공하고 Claude가 명확화 질문을 하는 단계
2. **개선 및 구조화**: 브레인스토밍과 편집을 통해 반복적으로 각 섹션을 구축하는 단계
3. **독자 테스트**: 새 Claude 인스턴스(맥락 없음)로 문서를 테스트하여 맹점을 발견하는 단계

이 접근법은 다른 사람이 읽을 때(Claude에 붙여넣는 경우 포함) 문서가 잘 작동하도록 보장한다고 설명합니다. 이 워크플로우를 사용할지 자유 형식으로 작업할지 물어봅니다.

사용자가 거부하면 자유 형식으로 작업합니다. 수락하면 Stage 1로 진행합니다.

## Stage 1: 컨텍스트 수집

**목표:** 사용자가 알고 있는 것과 Claude가 알고 있는 것 사이의 격차를 좁혀, 이후 단계에서 적절한 안내를 가능하게 합니다.

### 초기 질문

문서에 대한 메타 맥락을 사용자에게 물어봅니다:

1. 어떤 유형의 문서인가요? (예: 기술 사양서, 의사결정 문서, 제안서)
2. 주요 독자는 누구인가요?
3. 누군가 이 문서를 읽었을 때 원하는 영향은 무엇인가요?
4. 따라야 할 템플릿이나 특정 형식이 있나요?
5. 알아야 할 다른 제약 사항이나 맥락이 있나요?

짧게 답하거나 편한 방식으로 정보를 제공해도 된다고 안내합니다.

**사용자가 템플릿을 제공하거나 문서 유형을 언급한 경우:**
- 공유할 템플릿 문서가 있는지 물어봅니다
- 공유 문서 링크를 제공하면 적절한 통합 도구를 사용하여 가져옵니다
- 파일을 제공하면 읽습니다

**사용자가 기존 공유 문서 편집을 언급한 경우:**
- 적절한 통합 도구를 사용하여 현재 상태를 읽습니다
- alt-text 없는 이미지를 확인합니다
- alt-text 없는 이미지가 있으면, 다른 사람이 Claude를 사용하여 문서를 이해할 때 이미지를 볼 수 없다고 설명합니다. alt-text 생성을 원하는지 물어봅니다. 원하면 각 이미지를 채팅에 붙여넣어 설명적 alt-text를 생성하도록 요청합니다.

### 정보 덤프

초기 질문에 답변한 후, 사용자에게 가지고 있는 모든 맥락을 쏟아내도록 권유합니다. 다음과 같은 정보를 요청합니다:
- 프로젝트/문제의 배경
- 관련 팀 논의나 공유 문서
- 대안을 사용하지 않는 이유
- 조직적 맥락 (팀 역학, 과거 사건, 정치)
- 일정 압박이나 제약
- 기술 아키텍처나 의존성
- 이해관계자의 우려

정리하지 않아도 된다고 조언합니다 - 일단 모두 쏟아내라고 합니다. 맥락 제공 방법을 여러 가지 제안합니다:
- 의식의 흐름대로 정보 덤프
- 팀 채널이나 스레드 가리키기
- 공유 문서 링크

**통합 도구가 사용 가능한 경우** (예: Slack, Teams, Google Drive, SharePoint, 기타 MCP 서버), 이들을 사용하여 직접 맥락을 가져올 수 있다고 언급합니다.

**통합 도구가 감지되지 않고 Claude.ai 또는 Claude 앱인 경우:** Claude 설정에서 커넥터를 활성화하면 메시징 앱과 문서 저장소에서 직접 맥락을 가져올 수 있다고 제안합니다.

초기 덤프를 완료하면 명확화 질문을 할 것이라고 안내합니다.

**맥락 수집 중:**

- 사용자가 팀 채널이나 공유 문서를 언급하면:
  - 통합 도구 사용 가능: 지금 내용을 읽겠다고 안내한 후 적절한 통합 도구 사용
  - 통합 도구 사용 불가: 접근 불가를 설명합니다. Claude 설정에서 커넥터를 활성화하거나 관련 내용을 직접 붙여넣도록 제안합니다.

- 사용자가 알 수 없는 엔티티/프로젝트를 언급하면:
  - 연결된 도구로 검색하여 더 알아볼지 물어봅니다
  - 검색 전 사용자 확인을 기다립니다

- 사용자가 맥락을 제공할 때, 학습한 내용과 아직 불명확한 내용을 추적합니다

**명확화 질문:**

사용자가 초기 덤프를 완료했다고 신호를 보내면(또는 충분한 맥락이 제공된 후), 이해를 확인하기 위한 명확화 질문을 합니다:

맥락의 빈 부분을 기반으로 5-10개의 번호가 매겨진 질문을 생성합니다.

약식으로 답변하거나(예: "1: 예, 2: #channel 참조, 3: 하위 호환 때문에 아님"), 더 많은 문서 링크를 제공하거나, 채널을 가리키거나, 정보 덤프를 계속해도 된다고 안내합니다. 가장 효율적인 방식으로 하면 됩니다.

**종료 조건:**
질문이 이해도를 보여줄 때 충분한 맥락이 수집된 것입니다 - 기본 설명 없이 엣지 케이스와 트레이드오프에 대해 질문할 수 있을 때.

**전환:**
이 단계에서 더 제공할 맥락이 있는지, 아니면 문서 작성으로 넘어갈 때인지 물어봅니다.

더 추가하고 싶으면 그렇게 합니다. 준비되면 Stage 2로 진행합니다.

## Stage 2: 개선 및 구조화

**목표:** 브레인스토밍, 선별, 반복적 개선을 통해 섹션별로 문서를 구축합니다.

**사용자에게 안내:**
문서를 섹션별로 구축할 것이라고 설명합니다. 각 섹션마다:
1. 포함할 내용에 대한 명확화 질문
2. 5-20개 옵션 브레인스토밍
3. 사용자가 유지/제거/결합 표시
4. 섹션 초안 작성
5. 정밀한 편집을 통한 개선

불확실한 부분이 가장 많은 섹션부터 시작하고(보통 핵심 결정/제안), 나머지를 진행합니다.

**섹션 순서:**

문서 구조가 명확한 경우:
어떤 섹션부터 시작할지 물어봅니다.

불확실한 부분이 가장 많은 섹션부터 시작하도록 제안합니다. 의사결정 문서는 보통 핵심 제안, 사양서는 기술적 접근법입니다. 요약 섹션은 마지막에 남겨둡니다.

어떤 섹션이 필요한지 모르는 경우:
문서 유형과 템플릿에 기반하여 적절한 3-5개 섹션을 제안합니다.

이 구조가 괜찮은지, 조정하고 싶은지 물어봅니다.

**구조가 합의되면:**

모든 섹션에 대한 초기 문서 구조를 플레이스홀더 텍스트와 함께 생성합니다.

**아티팩트 접근이 가능한 경우:**
을 사용하여 아티팩트를 생성합니다. Claude와 사용자 모두에게 작업할 스캐폴드를 제공합니다.

모든 섹션에 대한 초기 구조를 플레이스홀더와 함께 생성하겠다고 안내합니다.

모든 섹션 헤더와 "[작성 예정]" 또는 "[내용 입력]"과 같은 간략한 플레이스홀더 텍스트로 아티팩트를 생성합니다.

스캐폴드 링크를 제공하고 각 섹션을 채울 시간이라고 안내합니다.

**아티팩트 접근이 불가능한 경우:**
작업 디렉토리에 마크다운 파일을 생성합니다. 적절한 이름을 부여합니다(예: , ).

모든 섹션에 대한 초기 구조를 플레이스홀더와 함께 생성하겠다고 안내합니다.

모든 섹션 헤더와 플레이스홀더 텍스트로 파일을 생성합니다.

파일명이 생성되었음을 확인하고 각 섹션을 채울 시간이라고 안내합니다.

**각 섹션마다:**

### 단계 1: 명확화 질문

[섹션명] 섹션 작업을 시작하겠다고 안내합니다. 포함할 내용에 대해 5-10개 명확화 질문을 합니다:

맥락과 섹션 목적에 기반하여 5-10개 구체적 질문을 생성합니다.

약식으로 답변하거나 다루어야 할 중요한 내용만 표시해도 된다고 안내합니다.

### 단계 2: 브레인스토밍

[섹션명] 섹션에 대해 섹션의 복잡도에 따라 [5-20]개 포함 가능 항목을 브레인스토밍합니다. 다음을 찾습니다:
- 잊었을 수 있는 공유된 맥락
- 아직 언급되지 않은 각도나 고려 사항

섹션 복잡도에 따라 5-20개 번호가 매겨진 옵션을 생성합니다. 마지막에 추가 옵션을 원하면 더 브레인스토밍하겠다고 제안합니다.

### 단계 3: 선별

어떤 포인트를 유지, 제거, 결합할지 물어봅니다. 다음 섹션을 위한 우선순위 학습을 위해 간략한 근거를 요청합니다.

예시 제공:
- "1,4,7,9 유지"
- "3 제거 (1과 중복)"
- "6 제거 (독자가 이미 알고 있음)"
- "11과 12 결합"

**사용자가 자유 형식 피드백을 제공하면** (예: "괜찮아 보입니다" 또는 "대부분 좋지만...") 번호 선택 대신, 선호도를 추출하고 진행합니다. 유지/제거/변경 사항을 파악하여 적용합니다.

### 단계 4: 갭 체크

선택한 내용을 기반으로, [섹션명] 섹션에서 빠진 중요한 내용이 있는지 물어봅니다.

### 단계 5: 초안 작성

를 사용하여 해당 섹션의 플레이스홀더 텍스트를 실제 초안 내용으로 교체합니다.

선택한 내용을 기반으로 [섹션명] 섹션을 초안 작성하겠다고 안내합니다.

**아티팩트 사용 시:**
초안 작성 후 아티팩트 링크를 제공합니다.

읽어보고 변경할 내용을 표시해달라고 요청합니다. 구체적일수록 다음 섹션을 위한 학습에 도움이 된다고 안내합니다.

**파일 사용 시 (아티팩트 없음):**
초안 작성 후 완료를 확인합니다.

[섹션명] 섹션이 [파일명]에 초안 작성되었다고 안내합니다. 읽어보고 변경할 내용을 표시해달라고 요청합니다. 구체적일수록 다음 섹션을 위한 학습에 도움이 된다고 안내합니다.

**사용자에게 핵심 안내 (첫 번째 섹션 초안 작성 시 포함):**
참고: 문서를 직접 편집하는 대신 변경할 내용을 표시해달라고 요청합니다. 이렇게 하면 향후 섹션을 위한 스타일 학습에 도움이 됩니다. 예시: "X 항목 제거 - Y에서 이미 다룸" 또는 "세 번째 단락을 더 간결하게".

### 단계 6: 반복적 개선

사용자가 피드백을 제공하면:
- 를 사용하여 편집합니다 (전체 문서를 다시 출력하지 않음)
- **아티팩트 사용 시:** 매 편집 후 아티팩트 링크 제공
- **파일 사용 시:** 편집 완료만 확인
- 사용자가 문서를 직접 편집하고 읽어달라고 요청하면: 사용자의 변경 사항을 내부적으로 기록하고 향후 섹션에 반영합니다 (선호도를 보여줌)

사용자가 섹션에 만족할 때까지 **반복을 계속**합니다.

### 품질 확인

연속 3회 반복에서 실질적인 변경이 없으면, 중요한 정보를 잃지 않으면서 제거할 수 있는 내용이 있는지 물어봅니다.

섹션이 완료되면 [섹션명]이 완료되었음을 확인합니다. 다음 섹션으로 넘어갈 준비가 되었는지 물어봅니다.

**모든 섹션에 대해 반복합니다.**

### 완성 단계

완성에 가까워지면 (80% 이상 섹션 완료), 전체 문서를 다시 읽고 다음을 확인하겠다고 안내합니다:
- 섹션 간 흐름과 일관성
- 중복이나 모순
- "슬롭"이나 일반적인 채우기처럼 느껴지는 부분
- 모든 문장이 의미 있는지

전체 문서를 읽고 피드백을 제공합니다.

**모든 섹션이 초안 작성되고 개선되면:**
모든 섹션이 초안 작성되었음을 안내합니다. 전체 문서를 한 번 더 검토하겠다고 합니다.

전체적인 일관성, 흐름, 완성도를 검토합니다.

최종 제안 사항을 제공합니다.

독자 테스트로 넘어갈 준비가 되었는지, 아니면 더 다듬을 부분이 있는지 물어봅니다.

## Stage 3: 독자 테스트

**목표:** 새 Claude 인스턴스(맥락 없음)로 문서를 테스트하여 독자에게 제대로 전달되는지 검증합니다.

**사용자에게 안내:**
문서가 실제로 독자에게 잘 작동하는지 테스트하겠다고 설명합니다. 이를 통해 맹점 — 저자에게는 이해되지만 다른 사람에게 혼란을 줄 수 있는 부분 — 을 발견합니다.

### 테스트 접근법

**서브 에이전트 접근이 가능한 경우 (예: Claude Code):**

사용자 개입 없이 직접 테스트를 수행합니다.

### 단계 1: 독자 질문 예측

독자가 이 문서를 발견하려 할 때 어떤 질문을 할지 예측하겠다고 안내합니다.

현실적으로 독자가 할 5-10개 질문을 생성합니다.

### 단계 2: 서브 에이전트로 테스트

이 대화의 맥락 없이 새 Claude 인스턴스로 질문을 테스트하겠다고 안내합니다.

각 질문에 대해 문서 내용과 질문만으로 서브 에이전트를 호출합니다.

독자 Claude가 각 질문에서 맞힌 것/틀린 것을 요약합니다.

### 단계 3: 추가 검사 실행

추가 검사를 수행하겠다고 안내합니다.

서브 에이전트를 호출하여 모호성, 잘못된 가정, 모순을 확인합니다.

발견된 이슈를 요약합니다.

### 단계 4: 보고 및 수정

이슈가 발견되면:
독자 Claude가 특정 이슈에서 어려움을 겪었다고 보고합니다.

구체적인 이슈를 나열합니다.

이 빈 부분을 수정하겠다고 안내합니다.

문제가 있는 섹션의 개선으로 돌아갑니다.

---

**서브 에이전트 접근이 불가능한 경우 (예: claude.ai 웹 인터페이스):**

사용자가 직접 테스트를 수행해야 합니다.

### 단계 1: 독자 질문 예측

사람들이 이 문서를 찾으려 할 때 어떤 질문을 할지 물어봅니다. Claude.ai에 무엇을 입력할까요?

현실적으로 독자가 할 5-10개 질문을 생성합니다.

### 단계 2: 테스트 설정

테스트 안내를 제공합니다:
1. 새 Claude 대화 열기: https://claude.ai
2. 문서 내용 붙여넣기 또는 공유 (커넥터가 활성화된 공유 문서 플랫폼이면 링크 제공)
3. 독자 Claude에게 생성된 질문 물어보기

각 질문에 대해 독자 Claude에게 다음을 제공하도록 지시합니다:
- 답변
- 모호하거나 불명확한 부분이 있는지
- 문서가 이미 알고 있다고 가정하는 지식/맥락

독자 Claude가 올바른 답변을 하는지 오해하는 부분이 없는지 확인합니다.

### 단계 3: 추가 검사

독자 Claude에게 추가로 물어봅니다:
- "이 문서에서 독자에게 모호하거나 불명확한 부분이 있나요?"
- "이 문서가 독자가 이미 알고 있다고 가정하는 지식이나 맥락은 무엇인가요?"
- "내부적인 모순이나 불일치가 있나요?"

### 단계 4: 결과에 따른 반복

독자 Claude가 틀리거나 어려움을 겪은 부분을 물어봅니다. 그 빈 부분을 수정하겠다고 안내합니다.

문제가 있는 섹션의 개선으로 돌아갑니다.

---

### 종료 조건 (두 접근법 모두)

독자 Claude가 일관되게 질문에 올바르게 답변하고 새로운 빈 부분이나 모호성을 제기하지 않으면 문서가 준비된 것입니다.

## 최종 검토

독자 테스트를 통과하면:
문서가 독자 Claude 테스트를 통과했다고 안내합니다. 완료 전에:

1. 본인이 최종 읽기를 한 번 하도록 권유합니다 - 문서의 소유자이며 품질에 대한 책임이 있습니다
2. 사실, 링크, 기술적 세부사항을 다시 확인하도록 제안합니다
3. 원했던 영향을 달성하는지 확인하도록 요청합니다

한 번 더 검토할지, 작업이 끝났는지 물어봅니다.

**최종 검토를 원하면 제공합니다. 그렇지 않으면:**
문서 완료를 안내합니다. 마지막 팁을 제공합니다:
- 독자가 문서 개발 과정을 볼 수 있도록 이 대화를 부록에 링크하는 것을 고려하세요
- 본문을 부풀리지 않으면서 깊이를 제공하기 위해 부록을 활용하세요
- 실제 독자로부터 피드백을 받으면 문서를 업데이트하세요

## 효과적인 안내를 위한 팁

**어조:**
- 직접적이고 절차적으로
- 사용자 행동에 영향을 미칠 때 근거를 간략히 설명
- 접근법을 "판매"하려 하지 않음 - 그냥 실행

**이탈 처리:**
- 사용자가 단계를 건너뛰고 싶어하면: 이 부분을 건너뛰고 자유 형식으로 작성할지 물어봅니다
- 사용자가 좌절한 것 같으면: 예상보다 오래 걸린다고 인정합니다. 더 빠르게 진행할 방법을 제안합니다
- 항상 사용자에게 프로세스 조정 권한을 줍니다

**맥락 관리:**
- 전체 과정에서 언급된 내용의 맥락이 부족하면 적극적으로 물어봅니다
- 빈 부분이 쌓이지 않도록 - 발생할 때마다 해결합니다

**아티팩트 관리:**
- 전체 섹션 초안 작성에  사용
- 모든 편집에  사용
- 매 변경 후 아티팩트 링크 제공
- 브레인스토밍 목록에는 아티팩트를 사용하지 않음 - 대화로 진행

**속도보다 품질:**
- 단계를 서두르지 않습니다
- 각 반복은 의미 있는 개선을 가져와야 합니다
- 목표는 독자에게 실제로 잘 전달되는 문서입니다
