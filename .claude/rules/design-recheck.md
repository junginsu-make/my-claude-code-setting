# Design Document & Existing Code Recheck Rule (CRITICAL)

## The Iron Law

```
매 phase/단계 시작 전, 설계 문서와 기존 코드를 반드시 재확인한다.
기억에 의존하지 않는다. 항상 파일을 다시 읽는다.
```

## 적용 범위

이 규칙은 코드 개선, 기능 구현, 버그 수정, 리팩토링 등
**모든 코드 변경 작업**에 적용된다. /auto 뿐만 아니라 수동 작업에도 적용.

## Phase/단계 시작 전 필수 체크리스트

### 1. 설계 문서 재확인
매 phase 시작 전 아래 문서가 존재하면 **반드시 다시 읽는다**:
- `prompt_plan.md` / `spec.md` — 구현 계획 및 요구사항
- `CLAUDE.md` / `AGENTS.md` — 프로젝트 규칙 및 컨벤션
- `docs/` 내 관련 설계 문서 — 아키텍처, DB 스키마 등
- 사용자가 제공한 요구사항 원문

### 2. 기존 코드 재확인
매 phase 시작 전 **수정 대상 파일과 관련 파일을 다시 읽는다**:
- 수정할 파일의 **현재 상태** (이전 phase에서 변경되었을 수 있음)
- import/export 관계에 있는 파일
- 같은 디렉토리의 관련 파일
- 테스트 파일 (존재하는 경우)

### 3. 변경 사항 대조
이전 phase에서 변경한 내용이 설계 문서와 일치하는지 확인한다:
- 설계 문서의 요구사항 vs 실제 구현 비교
- 누락된 요구사항이 없는지 체크
- 의도하지 않은 변경이 발생하지 않았는지 확인

## 금지 사항

| 금지 행위 | 올바른 행위 |
|-----------|------------|
| "아까 읽었으니 기억으로 진행" | 파일을 다시 Read 도구로 읽는다 |
| "설계 문서는 변하지 않았을 것" | 매 phase마다 다시 확인한다 |
| "이전 phase에서 잘 했으니 OK" | 현재 파일 상태를 다시 확인한다 |
| "간단한 변경이라 확인 불필요" | 모든 변경에 적용. 예외 없음 |

## Phase 전환 시 출력 형식

각 phase 시작 시 아래 형식으로 확인 결과를 남긴다:

```
[Phase N 시작] 설계 문서 & 기존 코드 재확인
  - 설계 문서: [파일명] 확인 완료
  - 수정 대상: [파일명] 현재 상태 확인 완료
  - 이전 phase 변경사항: 설계와 일치 확인
  → Phase N 진행
```

## 왜 이 규칙이 필요한가

LLM은 긴 작업에서 초기 맥락을 잃기 쉽다.
Phase가 진행될수록 설계 의도에서 벗어나는 "드리프트"가 발생한다.
매 phase마다 원본 설계를 재확인하면 이 드리프트를 방지할 수 있다.
